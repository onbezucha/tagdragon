## 🐛 Bugs

### Critical

- **`src/devtools/panel-bridge.ts:65-66`** — Iterace přes `heavyDataStore.keys()` se současným mazáním položek přes `originalDelete`. Ačkoliv `Map` obecně toleruje modifikaci za běhu iterace, specifikace nezaručuje konzistenci `keys()` iterátoru při souběžných `delete` operacích — může dojít k přeskočení položek nebo nekonečné smyčce. **Oprava:** Shromáždit klíče do pole před mazáním: `const toDelete = [...heavyDataStore.keys()].slice(0, deleteCount); toDelete.forEach(k => heavyDataStore.delete(k));`.

### Medium

- **`src/panel/datalayer/state.ts:58`** — Po auto-prune datalayer pushů se `clearDlPendingPushes()` zavolá bez pokusu o znovuzařazení ztracených pending push dat, která ještě nebyla renderována. Pending push data přijde o stav. **Oprava:** Přidat ztracené pending pushy zpět do fronty před vyčištěním, nebo zajistit, že `renderDlPushListFull()` je zpracuje.

- **`src/panel/components/detail-pane.ts:75`** — Non-null assertion `$detail!.classList` — pokud by `DOM.detail` byl `null` (např. chyba v HTML), dojde k runtime pádu bez varování. Podobné non-null assertions na mnoha místech (např. `DOM.envBadge!` v `adobe-env-switcher.ts:100`). **Oprava:** Přidat null-checky: `if (!$detail) return;`.

- **`src/panel/datalayer/utils/reverse-correlation.ts:37`** — `pushes[mid]._ts ?? 0` nahrazuje chybějící timestamp nulou, což posune push na začátek pole a může způsobit špatnou identifikaci triggeru při binárním vyhledávání. **Oprava:** Přeskočit položky bez validního `_ts`: přidat guard `if (midTime == null || isNaN(midTime)) { lo = mid + 1; continue; }`.

- **`src/background/popup-bridge.ts:26`** — `message.tabId` se používá bez validace na číslo. Pokud zpráva přijde s `tabId: undefined` nebo `tabId: "abc"`, výsledek `allStats[message.tabId]` bude chybný (přístup přes string klíč místo čísla). **Oprava:** Přidat validaci: `if (typeof message.tabId !== 'number' || message.tabId <= 0) return;`.

- **`src/panel/datalayer/components/push-detail.ts:400-402`** — Na řádcích 395-398 se přidávají childNodes do `container`, ale na řádku 401 se `container.innerHTML = '<div class="dl-state-empty">...'` přepíše celý obsah včetně předchozích `renderKvRow` položek. Podmínka `keys.length === 0` by měla být kontrolována PŘED přidáváním položek. **Oprava:** Přesunout `if (keys.length === 0)` check před `for` smyčku.

### Low

- **`src/content/data-layer-main.ts:412`** — `setInterval` s 500ms opakováním se spustí ihned a běží 20×. Pokud je `detectAndIntercept()` volána před inicializací `window.dataLayer`, pošle se prázdný replay (0 položek). Neškodné, ale zbytečné. **Oprava:** Přidat kontrolu `if (detected.gtm && Array.isArray(dl) && dl.length > 0)` před `replayDataLayer`.

## 🔒 Security

- **`src/panel/components/consent-panel.ts:319-325`** — **XSS zranitelnost:** `renderNoData(message)` vkládá parametr `message` přímo do `innerHTML` šablony bez escapování (`${message}`). Funkce je exportovaná a její budoucí použití s uživatelsky kontrolovaným vstupem by vytvořilo XSS vektor. **Oprava:** Nahradit `${message}` za `${esc(message)}`.

- **`src/panel/components/settings-drawer.ts:353-355`** — **XSS přes HTML atribut:** `value="${cfg.correlationWindowMs}"` vkládá hodnotu z configu do HTML atributu bez escapování. Pokud config obsahuje znaky jako `"`, dojde k injekci. **Oprava:** Použít `esc()` utilitu nebo `setAttribute('value', ...)`.

- **`src/background/index.ts:224-270`** — **CLEAR_COOKIES bez validace domény:** Handler přijímá URL z message body a parsuje ji. Jakákoliv message do background service workeru s typem `CLEAR_COOKIES` a libovolnou URL by smazala cookies pro danou doménu. **Oprava:** Přidat validaci `tabId` do zprávy a ověřit, že URL odpovídá doméně dané karty přes `chrome.tabs.get(tabId)`.

- **`src/content/data-layer-bridge.ts:20-21`** — **postMessage bez důvěryhodného channel:** Bridge naslouchá na `TAGDRAGON_DL_PUSH` zprávy bez dodatečné validace. Jakákoliv stránka na stejné doméně může odeslat zprávu s tímto typem a bridge ji přepošle. Pro DevTools nástroj je to přijatelné riziko, ale worth noting. **Oprava:** Přidat validaci na specifický message channel nebo tajný token.

- **`manifest.json:23-28`** — **Široká oprávnění:** `webRequest`, `cookies`, `scripting`, `<all_urls>` host permissions. Pro nástroj DevTools nutné, ale `webRequest` umožňuje číst veškerý síťový provoz. Chrome Web Store audit by to mohl označit. **Oprava:** Zvážit použití `declarativeNetRequest` místo `webRequest` pro `onBeforeSendHeaders` listener, ale vyžadovalo by to refaktoring capture logiky.

## ⚠️ Architecture Concerns

- **Smíšená komunikace DevTools ↔ Panel přes globální `window` funkce** — `panel/index.ts` definuje vlastní `declare global { interface Window }` blok (ř. 123-140), který se částečně překrývá s `PanelWindow` v `panel-bridge.ts`. Rozdělení typů do dvou míst ztěžuje údržbu. **Oprava:** Centralizovat všechny `window` rozšíření do jednoho sdíleného typu v `src/types/panel-window.ts`.

- **Monkey-patching `Map.prototype` v panel-bridge** — `heavyDataStore` má přepsané metody `set`, `delete` a `clear` pro sledování velikosti a automatickou evikci (ř. 56-90). Porušuje princip nejmenšího překvapení a znemožňuje vytvořit novou instanci s tímto chováním. **Oprava:** Vytvořit třídu `SizeTrackingMap<K,V>` zapouzdřující Map s vlastním `set/delete/clear`.

- **Závislost DataLayer stavu na Network stavu** — `src/panel/datalayer/state.ts:11` přímo importuje `getAppConfig` a `updateConfigImmediate` z `@/panel/state`. DataLayer modul nelze testovat nebo používat bez inicializace network stavu. **Oprava:** Zvážit injektování config callbacku místo přímého importu.

- **`devtools/index.ts:100-106`** — unsafe `window` přístup přes `(win as unknown as Record<string, unknown>)['_setPaused']` místo využití `PanelWindow` interface. Ztráta type safety. **Oprava:** Rozšířit `PanelWindow` interface a použít typový cast.

- **`popover-manager.ts` neobsahuje odstraňovací funkci** — Popover lze registrovat (`registerPopover`) a zavřít všechny (`closeAllPopovers`), ale chybí funkce `unregisterPopover(name)`. Pokud se popover odstraní z DOM, jeho close handler zůstane v mapě. **Oprava:** Přidat `unregisterPopover`.

## ⚡ Performance

### Kritické

- **`src/panel/components/provider-bar.ts:187-191`** — `ensureProviderPill()` volá 5 DOM update funkcí po každém novém requestu. Při vysokém trafficu (desítky requestů/s) způsobuje stovky reflow/repaint cycle. **Oprava:** Sloučit do jedné funkce `syncProviderBar()` volané jednou za request batch.

- **`src/panel/datalayer/utils/correlation.ts:30-42`** — `findCorrelatedRequests` iteruje přes VŠECHNY network requests pro každý DataLayer push. S 500 requests a 1000 pushi = 500 000 porovnání na jednu korelaci. **Oprava:** Budovat index network requests podle času (`Map<timestamp, request>`) při jejich přidávání; použít binární hledání.

### Střední

- **`src/panel/datalayer/components/push-list.ts:219, 243`** — `navigateDlList` a `navigateDlToEdge` volají `pushes.find()` — O(n) lineární hledání při každé navigaci šipkami. **Oprava:** Použít `dlState.getDlPushById(pushId)` — O(1) Map lookup.

- **`src/panel/components/filter-bar.ts:179-244`** — `updateActiveFilters()` vytváří nové DOM elementy pro každý chip včetně innerHTML concatenation. Navíc rekurzivně volá sama sebe v `onRemove` callbacku (double-render). **Oprava:** Použít DocumentFragment pro batch DOM append; odstranit rekurzivní volání.

- **`src/panel/datalayer/utils/correlation.ts:86, 91`** — `renderCorrelation` volá `tryParseUrl()` pro KAŽDÝ korelovaný request v render path. S 50+ korelovanými requesty = 50+ `new URL()` konstrukcí. **Oprava:** URL parsing provést v `findCorrelatedRequests` a uložit do objektu.

- **`src/panel/datalayer/components/push-list.ts:403-431`** — `getSortedDlPushIds()` vytváří vždy nové pole `[...all].reverse()` nebo `[...all].sort()` pro každé volání. S 1000 pushi a častým scrollováním = opakované allocace. **Oprava:** Výsledky cachovat pomocí `Map<string, number[]>` pod klíčem `${field}-${order}`; invalidovat při prune.

- **`src/panel/detail-tabs/response.ts:14-18`** — `JSON.parse()` následované `JSON.stringify()` pro každý response tab render. Při velkých JSON odpovědích (500KB+) pomalé a paměťově náročné. **Oprava:** Provést parse pouze pokud `body.length < 10000`; jinak zobrazit surový text s oříznutím.

- **`src/panel/datalayer/components/push-detail.ts:365-379`** — `renderDiffTab` volá `computeCumulativeState()` při každém otevření Diff tab — O(n²) pro n-tý push bez cache. **Oprava:** Ověřit že všechny pushi mají `cumulativeState`; použít `Map<pushIndex, state>` cache.

- **`src/panel/utils/categorize.ts:74-84`** — `categorizeParams` prochází všechny regex patterns pro každý parametr. S 50+ parametry = stovky `pattern.test()` volání na request. **Oprava:** Prefix match a regex patterns předkompilovat do jedné lookup struktury.

- **`src/panel/datalayer/components/live-inspector.ts:151`** — `JSON.stringify(oldVal) !== JSON.stringify(newVal)` se volá pro KAŽDOU změnu na každé sledované cestě. S 10 watch paths a častými updates = mnoho serializací. **Oprava:** Použít custom deep-equal funkci bez serializace.

- **`src/panel/utils/dom.ts:98-303`** — Getter-based DOM refs (`get list()`) provádí `document.getElementById` při každém přístupu. V tight loopách (request-list rendering) může být voláno 100x+ za frame. **Oprava:** Pro často používané refs definovat konstanty při init; ostatní nechat jako lazy getters.

- **`src/providers/url-parser.ts:64-72`** — `getParams` se pokouší `JSON.parse(bodyStr)` pro KAŽDÝ POST request. Většina POST dat jsou urlencoded stringy, ne JSON. **Oprava:** Nejprve zkusit URLSearchParams; JSON parsing pouze jako fallback.

### Nízké

- **`src/panel/datalayer/components/push-list.ts:139-161`** — `buildPreview` volá `typeof v === 'object'` kontrolu 3x na jednu hodnotu. **Oprava:** Proměnná `isObj = typeof v === 'object' && v !== null` definovaná jednou.

- **`src/panel/detail-tabs/decoded.ts:25-63`** — Template string concatenation `html += ...` v loopi vytváří mnoho intermediate stringů. **Oprava:** Použít pole `const parts: string[] = []` a `parts.join('')`.

- **`src/panel/datalayer/state.ts:125-127`** — `getDlSourceCount` provádí `filter` přes všechny pushi pro každé volání. **Oprava:** Počítadla udržovat inkrementálně — `Map<source, count>`.

- **`src/panel/components/provider-bar.ts:46-50`** — `ensureProviderGroup` kontroluje existenci group pomocí `querySelector` před vytvořením. **Oprava:** Udržovat `Map<groupId, HTMLElement>` v paměti.

## ✅ Shrnutí

TagDragon je dobře strukturovaný a udržovaný Chrome Extension s čistým typovým systémem a konzistentními konvencemi. Hlavní oblasti pro zlepšení jsou: (1) **výkonnost korelačního enginu** — O(n×m) složitost je neudržitelná při vysokém trafficu, (2) **velikost panel controlleru** (`panel/index.ts` s 1613 řádky) porušuje SRP a ztěžuje orientaci, (3) **několik XSS vektorů** v `consent-panel.ts` a `settings-drawer.ts` vyžaduje okamžitou opravu, a (4) **duplicitní logika** (barvy zdrojů, `getNestedValue`, filter chips) by měla být konsolidována.

### Prioritní pořadí
1. 🔒 Opravit XSS zranitelnosti v `consent-panel.ts` a `settings-drawer.ts`
2. ⚡ Optimalizovat korelační engine (časový index místo O(n×m) brute force)
3. ♻️ Extrahovat duplicitní logiku do sdílených modulů (`getNestedValue`, barvy zdrojů, `MAX_BUFFER`)
4. 🐛 Opravit iteraci `Map.keys()` souběžně s mazáním v `panel-bridge.ts`
5. ⚠️ Zmenšit `panel/index.ts` extrakcí DataLayer filter chips a splitter logiky
