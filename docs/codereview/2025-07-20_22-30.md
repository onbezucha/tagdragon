## 🐛 Bugs

### Critical
None found.

### Medium

- **`src/devtools/network-capture.ts:100,128`** — Request duration calculated incorrectly. `Math.round(req.time)` treats the value as milliseconds, but `chrome.devtools.network.Request.time` returns **seconds** (floating-point). All fast requests (< 500ms) show `0ms`, and others display wrong durations (e.g., 1.5s request shows as `2ms` instead of `1500ms`). **Fix:** Change `Math.round(req.time)` to `Math.round(req.time * 1000)` on both lines.
  **Reproduce:** Open any website with analytics tracking in DevTools with TagDragon. Most requests show `0ms` duration in the detail pane while Chrome's Network tab shows non-zero values.

- **`src/devtools/network-capture.ts:192-200`** — `RECORDING_PAUSED`/`RECORDING_RESUMED` messages check `msg.tabId === tabId`, but `popup-bridge.ts:183` sends these via port `postMessage` **without `tabId`**. Since `chrome.runtime.onMessage` and port `onMessage` are separate channels, this handler **never fires**. The local `isPaused` flag is never updated at runtime — it stays at the initial value loaded from storage. While the panel UI correctly stops displaying requests, the network capture continues processing all requests (calling `matchProvider`, `getParams`, `req.getContent`, heavy data storage) even when paused, wasting CPU and memory. **Fix:** Remove the `msg.tabId === tabId` checks from lines 192 and 197, since `chrome.runtime.onMessage` in DevTools context only receives messages addressed to the extension. Alternatively, forward pause state from `devtools/index.ts` via a direct function call.
  **Reproduce:** 1) Open DevTools with TagDragon on any page. 2) Click the extension icon → Pause. 3) Panel stops showing new requests, but network-capture continues processing all requests in the background.

- **`src/content/data-layer-main.ts:119-124`** — The monkey-patched `dataLayer.push` assigns the same `pushIndex` to all arguments when called with multiple args (e.g., `dataLayer.push({event:'x'}, {event:'y'})`). Since `dataLayer.length` hasn't been updated yet (the original `push` is called after `sendPush`), all args get the same index. Same issue for `adobeDataLayer.push` at line 213-218. **Fix:** Use a counter offset: `sendPush('gtm', dataLayer.length + i, ...)` where `i` is the arg index in the `forEach` callback.
  **Reproduce:** In browser console on a page with GTM, execute `dataLayer.push({event:'test1'}, {event:'test2'})`. In TagDragon's DataLayer tab, both pushes show the same push index.

- **`src/panel/datalayer/state.ts:65`** — `addDlPush()` clears `dlState.filteredIds` on prune but doesn't rebuild the filtered set. The caller (`panel/index.ts:754-755`) immediately calls `dlApplyFilter()` + `renderDlPushListFull()`, but the function leaves filtered IDs in an inconsistent state between return and caller fixup. If any code reads `getDlFilteredIds()` between the prune return and `dlApplyFilter()`, it would see an empty set. **Fix:** Remove `dlState.filteredIds.clear()` from `addDlPush` and let the caller handle it exclusively (which it already does), or rebuild the filtered set inside `addDlPush` before returning.

### Low

- **`src/panel/index.ts:375`** — Uses `innerHTML` instead of `textContent` for datalayer hint text. While the string contains no user-controlled data, `textContent` would be more consistent with line 377 which already uses `textContent`. **Fix:** Change `innerHTML` to `textContent`.

- **`src/background/popup-bridge.ts:171-173`** — Dead code: `const targetTabId = tabId ?? (await getActiveTabId())`. The function returns early if `tabId` is not a valid positive number, so `tabId` is always a valid `number` at this point. The `??` fallback and unnecessary `await getActiveTabId()` call are dead code. **Fix:** Use `tabId` directly.

## 🔒 Security

- **`src/background/index.ts:224-226`** — The `CLEAR_COOKIES` handler accepts a `url` from the message body without verifying it belongs to the sender's tab. A compromised or malicious DevTools panel could send `CLEAR_COOKIES` with an arbitrary URL to delete cookies for any domain. **Fix:** Validate that the URL's origin matches `_sender.tab?.url` origin:
  ```typescript
  if (_sender.tab?.url) {
    try {
      const senderOrigin = new URL(_sender.tab.url).origin;
      const targetOrigin = new URL(url).origin;
      if (senderOrigin !== targetOrigin) {
        sendResponse({ deleted: 0 });
        return true;
      }
    } catch {
      sendResponse({ deleted: 0 });
      return true;
    }
  }
  ```
  **Reproduce:** Send `chrome.runtime.sendMessage({ type: 'CLEAR_COOKIES', url: 'https://example.com' })` from a DevTools panel inspecting a different origin — cookies for `example.com` would be deleted.

## ♻️ Code Quality

- **`src/panel/index.ts`** (1618 lines) — Massive SRP violation. Serves as: network request receiver, DataLayer push receiver, DataLayer filter logic, DataLayer filter chip rendering, batched rendering for both tabs, view switching, toolbar handlers, quick actions, export logic, category toggle, copy handler, request list click handler, DataLayer full list renderer, DataLayer handler initialization, pause UI sync, CSV export, and init function. Should be split into at least 3-4 focused modules (network receiver, DataLayer controller, view switching, init only).

- **`src/panel/components/settings-drawer.ts`** (792 lines) — Combines settings UI rendering, section accordion logic, control wiring, validation rule UI management (lines 578-777), config import/export, and settings search. The validation rule UI should be extracted to a dedicated module.

- **`src/panel/index.ts:716-785`** — `receiveDataLayerPush()` mixes data enrichment, cumulative state mutation, filtering, batching, prune handling, and DOM badge updates in a single callback with 6 distinct responsibilities. The prune branch (lines 744-766) duplicates badge update logic from `flushPendingDlPushes`.

- **`src/panel/components/settings-drawer.ts:176` and `src/panel/components/dl-filter-popover.ts:156`** — Identical `closeAllExpands()` functions defined in two files. Should be extracted to a shared utility.

- **`src/panel/components/settings-drawer.ts:213` and `src/panel/components/dl-filter-popover.ts:22`** — Identical `CHEVRON_SVG` constants defined in two files. Should live in a shared UI constants module.

- **`src/panel/index.ts:1139-1150`** — A single `document.addEventListener('click', ...)` closes three popovers (info, provider filter, DL filter). Meanwhile `adobe-env-switcher.ts:353` and `consent-panel.ts:106` each register their own separate click handlers for close-on-outside-click. The settings drawer uses `closeAllPopovers()` from the popover manager. Inconsistent close-on-outside-click architecture.

- **`src/panel/index.ts:55` and `src/panel/datalayer/components/push-list.ts:12`** — Network filter chips (`filter-bar.ts`) and DataLayer filter chips (`panel/index.ts`) implement the same UX concept (removable filter pills) with completely different DOM structures, classes, and APIs. Should share a common filter chip rendering utility.

- **`src/panel/components/settings-drawer.ts:207`** — Helper function `nd()` is overly terse. Its purpose is "returns a CSS class if config value differs from default." A name like `nonDefaultClass()` would be clearer.

- **`src/panel/index.ts:1051,1343`** — Filter debounce timer `150` ms used in two places as a magic number. Should be a named constant like `FILTER_DEBOUNCE_MS`.

- **`src/panel/index.ts:1120`** — Cookie clear feedback timeout `2000` ms is a magic number. Should be named.

- **`src/devtools/panel-bridge.ts:146-162`** — `PanelWindow` interface duplicates the Window extension from `panel/index.ts:125-142`. Both define the same methods but in separate locations. If one is updated, the other can drift silently. Should be a single shared type.

- **`src/panel/datalayer/state.ts:30-36,307-308,358-361`** — `DlFilterState`, `DlSortField`, `DlSortOrder`, and `DlPendingPush` types are defined locally but should be in `src/types/datalayer.ts` alongside other DataLayer types.

## ⚠️ Architecture Concerns

- **`src/panel/index.ts` at 1618 lines is the biggest architectural risk.** It mixes network controller, DataLayer controller, filter logic, rendering, and UI event handling. Splitting this into focused modules would be the single highest-impact improvement. Affected files: `src/panel/index.ts` (+ all files that import from it).

- **DataLayer state couples to network state for config persistence.** `src/panel/datalayer/state.ts:11` directly imports `getAppConfig` and `updateConfigImmediate` from `src/panel/state.ts`, preventing independent testing and reuse. Affected files: `src/panel/datalayer/state.ts`, `src/panel/state.ts`.

- **Background message handler is monolithic.** `src/background/index.ts:33-271` — The `chrome.runtime.onMessage` listener is a 240-line handler containing all message routing logic (Adobe redirect CRUD, DataLayer relay, cookie clearing, script injection). Each message type is a distinct domain crammed into one callback. Affected file: `src/background/index.ts`.

- **Popover close-on-outside-click is inconsistent.** Some popovers use the popover manager pattern, others register their own click handlers. No unified approach. Affected files: `src/panel/index.ts`, `src/panel/components/adobe-env-switcher.ts`, `src/panel/components/consent-panel.ts`, `src/panel/utils/popover-manager.ts`.

- **Network and DataLayer filter chip systems are separate implementations** despite serving the same UX purpose. Different DOM structures, classes, and APIs. Affected files: `src/panel/components/filter-bar.ts`, `src/panel/index.ts`.

## ⚡ Performance

- **`src/panel/state.ts:340-342,457-459`** — `getStats()` and `getConfig()` spread state objects on every call (`return { ...statsState }` / `return { ...config }`), creating new object allocations. Called from hot paths (row creation, filter checks, status bar updates). **Fix:** Return the state object directly for internal consumers, or use a frozen/cached copy. **Impact:** ~50-100 object allocations per second during active capture.

- **`src/panel/components/provider-bar.ts:219-244`** — `updateProviderCounts()` iterates all requests to build counts, then for every group iterates all pills to compute group totals — O(providers × groups × requests). Called on every new request. **Fix:** Maintain provider counts incrementally as requests are added. **Impact:** At 500 requests, re-scans all data repeatedly (~5ms per batch flush).

- **`src/panel/datalayer/utils/correlation.ts:30-42`** — `findCorrelatedRequests` calls `new Date().getTime()` up to 3 times per request (lines 27, 32, 40). **Fix:** Parse `push._ts` once, compute `r._ts` once per request. **Impact:** 1500 unnecessary Date objects for 500 requests per correlation call.

- **`src/panel/datalayer/utils/reverse-correlation.ts:53-65`** — Post-binary-search linear scan iterates all pushes from `lo` to end without breaking when `delay > lookbackMs`. **Fix:** Break early when `delay > lookbackMs`. **Impact:** O(n) instead of bounded O(lookback window) in worst case.

- **`src/panel/utils/filter.ts:10`** — `_regexCache` Map grows unbounded as users enter new regex filters. No eviction policy. **Fix:** Add LRU eviction (cap at 50 entries). **Impact:** Minor memory growth for long sessions.

- **`src/content/data-layer-main.ts:408-414`** — `setInterval` with 500ms retry continues running even after all sources are detected. **Fix:** `clearInterval` once all sources detected. **Impact:** Up to 20 unnecessary interval callbacks.

- **`src/background/index.ts:177-221`** — `INJECT_DATALAYER` handler runs 4 sequential `chrome.scripting.executeScript` calls with `await` (~200-400ms total). **Fix:** Run guard-clearing steps in parallel. **Impact:** ~100ms saved per injection.

- **`src/panel/utils/session-persist.ts:23-29`** — Session storage saves debounced at 500ms. At 10 requests/sec = 5 saves/sec. **Fix:** Increase debounce to 1000-2000ms. **Impact:** Halves sessionStorage write frequency.

## ⚠️ Accepted Risks

Issues identified but NOT recommended to fix — the fix would add more complexity than value, or the current approach is acceptable.

- **`src/panel/components/request-list.ts:125-160`** — No virtualization for request list (max 500 rows). At ≤500 rows, DOM performance is acceptable. Adding virtualization would add significant complexity for marginal benefit. **Status:** ACCEPTED RISK — revisit if `maxRequests` is increased.

- **`src/panel/datalayer/components/push-list.ts:404-438`** — `getSortedDlPushIds()` creates intermediate arrays during sort+map. The allocation overhead (~0.5ms per sort for 1000 pushes) is negligible. **Status:** ACCEPTED RISK.

- **`src/providers/index.ts:243-298`** — `matchProvider` may test 2-4 duplicate candidates for popular domains. The overhead is minimal (a few extra regex tests). **Status:** ACCEPTED RISK — deduplication would add Set allocation overhead that may exceed the savings.

- **`src/panel/datalayer/utils/diff-renderer.ts:71`** — `JSON.stringify` fallback in diff algorithm. The fast paths (primitives, reference equality) are exercised first; the stringify fallback only hits on complex objects. **Status:** ACCEPTED RISK.

- **`src/panel/datalayer/utils/validator.ts:223-233`** — `hasUndefinedValues` recursion with `depth > 5` guard. For deeply nested GTM data this may not go deep enough, but increasing the limit risks stack overflow on pathological data. **Status:** ACCEPTED RISK — depth 5 is a reasonable safety threshold.

- **`src/shared/datalayer-constants.ts:6-25`** — `SOURCE_LABELS` and `SOURCE_DESCRIPTIONS` have minor redundancy (2 identical entries out of 6). Consolidation is non-trivial since they serve different UI roles. **Status:** ACCEPTED RISK.

## ✅ Summary

TagDragon is a well-structured Chrome extension with clean TypeScript, consistent conventions, and thoughtful performance patterns (RAF batching, template cloning, icon caching, lazy loading). The codebase has zero TODO/FIXME/HACK debt and passes strict TypeScript checks. The main concerns are: (1) a duration calculation bug that shows wrong request timings, (2) the main panel controller being 1618 lines with mixed responsibilities, and (3) a paused-recording bypass where network capture continues processing even when the user hits pause. No critical bugs or exploitable security vulnerabilities were found.
